## Decisions I made

- **Stayed on Next 15.5.15 (the `backport` channel) rather than bumping to Next 16.2.4.** Next 16 is `latest` on npm, but `CLAUDE.md` and the brief explicitly name "Next.js 15" as the stack decision; the Next 16 jump is a stack-level call that belongs to Cowork, not me. The 15.x backport line is actively maintained, so we're current within the chosen major. Same logic for **Tailwind**: kept on `^3.4.19` (the `v3-lts` tag) instead of bumping to `4.2.4` because v4 changes the CSS pipeline (no `postcss.config.mjs`, `@import "tailwindcss"` instead of `@tailwind base;`) and that migration deserves its own brief — especially since Phase 2 will start porting prototype CSS into this pipeline.
- **Switched `lint` from `next lint` → `eslint .`.** `next lint` prints a deprecation warning and will be removed in Next 16. Calling `eslint` directly works under the flat config we already need for ESLint 9 and survives the eventual Next 16 upgrade with no further changes.
- **Used `FlatCompat` with the legacy `next/core-web-vitals` + `next/typescript` configs** rather than a hand-rolled flat config. That's the pattern `eslint-config-next@15.5.x` ships with — the package still publishes the legacy `index.js` and `typescript.js` rather than a native flat-config entry. `@eslint/eslintrc` is already a transitive dep of ESLint 9, so no new dependency.
- **Kept the architect's `<link>`-tag font loading** in `src/app/layout.tsx` despite the lone `@next/next/no-page-custom-font` warning. The architect explicitly chose this in the file's docblock ("five families is too much for next/font self-hosting; switch when we settle on 2–3 weights"). Lint passes (0 errors); the warning is intentional.
- **Single Supabase URL for runtime + migrations** instead of split URLs. The brief's open question 4 assumed direct-connection for migrations; that doesn't work — see Open issues. A single Transaction-pooler URL on port 6543 with `prepare: false` handles both layers. drizzle-kit's migrator works fine through it; no advisory locks involved. If we later need to tune for runtime perf or run schema-altering migrations that the transaction pooler can't handle, splitting into `DATABASE_URL` (transaction, runtime) + `MIGRATE_DATABASE_URL` (session, migrations) is a one-line `drizzle.config.ts` change.
- **Fixed `seed.ts`'s env-loading via `tsx --env-file`** rather than reordering imports inside the file. ESM hoists all `import` statements above any executable code, so the architect's `loadEnv({ path: ".env.local" })` call could never run before `import { db } from "@/db/client"` evaluated and threw. `--env-file` (Node 20.6+, supported by tsx) sidesteps the whole problem and removes a runtime dependency on `dotenv` for the seed path.
- **URL-encoded the database password** (`*kxLf4g,!hx2BNF` → `%2AkxLf4g%2C%21hx2BNF`) in `DATABASE_URL`. `postgres-js` parses the URI before reaching the password, so raw `*`/`,`/`!` are technically valid sub-delims but bite some parsers; encoding is cheap insurance. Documented in `.env.local` (gitignored) and noted in `.env.example` / `ONBOARDING.md`.
- **Stripped the `[…]` brackets** from the password Philipp pasted. They were leftovers from Supabase's `[YOUR-PASSWORD]` placeholder syntax in the dashboard, not part of the password.
