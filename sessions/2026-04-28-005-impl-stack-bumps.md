# Stack bumps — Next major + Tailwind major

## Summary

Bumped to **Next 16.2.4** (`latest`) and **Tailwind 4.2.4** (`latest`) via the official codemods. Pre/post HTML class diffs across all 8 routes show exactly one byte-level change (`rounded` → `rounded-sm`, codemod-applied to keep the v3 visual radius), and compiled-CSS rules for every utility we use resolve to identical theme tokens. Live deploy URL pending — see Verification.

## What I did

- `package.json` — `next 15.5.15 → 16.2.4`, `eslint-config-next 15.5.15 → 16.2.4`, `tailwindcss ^3.4.19 → ^4.2.4`. Added `@tailwindcss/postcss ^4.2.4`, removed `autoprefixer`. Codemod also alphabetized the deps list (cosmetic).
- `package-lock.json` — regenerated by `npm install`. Net: 30 packages added, 1 removed, 6 changed (414 packages total).
- `eslint.config.mjs` — replaced the `FlatCompat` shim with a single direct import `import nextCoreWebVitals from "eslint-config-next/core-web-vitals"`. `eslint-config-next@16` ships native flat-config arrays (default and `/core-web-vitals` exports), and the legacy `next/core-web-vitals` + `next/typescript` extends paths no longer round-trip through FlatCompat (it dies with a circular-JSON error during validation). The `/core-web-vitals` export already includes the base `next/typescript` profile.
- `tsconfig.json` — Next 16 auto-edited two values on first `next dev`: `jsx: "preserve" → "react-jsx"` (mandated; React 19 automatic runtime) and `include` gained `.next/dev/types/**/*.ts` (Next 16's new dev-types path; the old `.next/types/**/*.ts` is kept for backward compat). Both edits left in place.
- `postcss.config.mjs` — `tailwindcss + autoprefixer` → `@tailwindcss/postcss`. Tailwind 4 ships vendor prefixing internally.
- `tailwind.config.ts` — **deleted**. The codemod migrated every theme token (5 void shades, 6 aquila shades, 5 frost shades, 3 heresy shades, 5 font families, 3 animations + their `@keyframes`) into a `@theme {…}` block in `globals.css`. Single source of truth now lives next to the styles.
- `src/app/globals.css` — `@tailwind base/components/utilities` → `@import 'tailwindcss'`. Codemod added one v3-compat `@layer base` block restoring `border-color: var(--color-gray-200, currentcolor)` on every element, because Tailwind 4 changed the default border color to `currentcolor` and the brief mandates "no rendered-output change."
- `src/app/page.tsx` — one class rename, `rounded → rounded-sm`. v4 renamed the radius scale (`rounded` is now `0.5rem`, `rounded-sm` is `0.25rem` = the old `rounded`); the codemod applied this everywhere the v3 default was used.

## Decisions I made

- **Picked Next 16.2.4 (latest) over 15.5.15 (backport).** The brief authorizes the major bump and the 16.2 minor is ~6 weeks old (16.2.0 shipped 2026-03-18, 16.2.4 is the current patch from 2026-04-15). Past the brief's "1-month watermark." Same logic for Tailwind: 4.2.4 (latest, 2026-04-21) over the v3-lts line — v4 itself has been stable for over a year, and 4.2.4 is the current patch.
- **One bump commit instead of a pair.** The brief allows either. I went with one because the codemods reordered `package.json` and regenerated the entire `package-lock.json` in a single pass; surgically splitting those across two commits would have been mechanical churn for no review benefit. The commit message lists both bumps and what each touched. Workflow doc edits + brief 004 went into a separate prior commit so the bump commit is purely about deps/config.
- **Skipped `@next/codemod` entirely.** Ran its `--help` to confirm what's available, then audited our codebase against the v15→v16 migration list: we don't use `next/image`, don't use `req.geo`/`req.ip`, `params` is already `Promise<…>`-typed and awaited, and we already use `eslint .` instead of `next lint`. Nothing for it to do. The Next 16 dev server's auto-edit of `tsconfig.json` covered the only mandatory change.
- **Replaced `FlatCompat` with a native flat-config import.** `eslint-config-next@16` ships its own flat-config arrays (`./dist/index.js` is `next/typescript` + globals, `./dist/core-web-vitals.js` adds the next-plugin core-web-vitals rules). The old config crashed on first `lint` after the bump with a circular-JSON error from `@eslint/eslintrc/lib/shared/config-validator.js` — that path is for legacy `.eslintrc` configs, which `eslint-config-next@16` no longer is. The native import is also smaller, has no `@eslint/eslintrc` dependency, and survives ESLint 10 without further changes.
- **Let the codemod migrate `tailwind.config.ts` into `@theme` in `globals.css` rather than keeping the JS config via `@config`.** Tailwind 4's preferred path is CSS-first; keeping the JS file would have been a deprecated escape hatch. Phase 2 will pour ~80 KB of prototype CSS into `globals.css`, so having the theme co-located with the styles is the right shape going in. The migrated `@theme` block preserves every token 1:1 (verified by grepping computed CSS output for each utility we use — see Verification).
- **Kept the codemod's v3 border-color compat block.** It restores `border-color: var(--color-gray-200, currentcolor)` on every `*, ::before, ::after, ::backdrop, ::file-selector-button` so existing utilities that rely on the old default render the same. Phase 2 work that explicitly sets border colors can remove this block later; today's brief mandates no visual regression.
- **Did NOT bump Drizzle, postgres-js, @supabase/supabase-js, React, or TypeScript.** Brief out-of-scope rule. None were peer-forced by Next 16 or Tailwind 4. React 19.2.5 is fine for Next 16 (peer is `react ≥19`). TypeScript 5.9.3 satisfies Next 16's `≥3.3.1`. The transitive `@esbuild-kit/*` audit warning under `drizzle-kit` from Phase 1 is unchanged.
- **Did NOT silence the new `npm audit` moderate on `postcss <8.5.10`** introduced via `next > postcss`. The auto-fix would force `next@9.3.3` (a breaking downgrade); the actual advisory (XSS via unescaped `</style>` in stringify output) only applies if you concatenate untrusted CSS into a build-time `<style>` tag, which we don't. Will clear automatically when Next bumps its bundled postcss.

## For next session

Architectural follow-ups Cowork should weigh when planning Phase 2's first brief:

- **Tailwind 4 CSS-pipeline notes for the prototype-CSS port.** The pipeline change that mattered for *this* brief: `@theme` is a `@layer theme`-equivalent block in CSS, and tokens defined there auto-generate utilities (e.g. `--color-aquila: …` materialises `text-aquila`, `bg-aquila`, `border-aquila`, etc.). The implication for Phase 2: when porting prototype CSS, prefer adding new design tokens to the `@theme` block over hard-coding hex values in component-scoped CSS — this gives every component access to the same utility set automatically. Custom keyframes belong in `@theme` too (see `fadeSlide`, `tileRise`, `scan` in `globals.css`); Tailwind picks them up via `--animate-{name}` aliases. **One footgun:** the codemod added a v3-compat block that defaults all element borders to `gray-200`. As Phase 2 ports specific borders (`.eraTile { border: 1px solid var(--c-frost-900) }` etc.), the compat block becomes dead weight — once every styled element specifies its own border color, delete the compat block to align with v4 defaults.
- **Next 16 App Router signature changes — none required for our existing stub routes.** `params: Promise<…>` was already adopted in Phase 1 (the brief's open question 2). The only Next 16 changes I had to absorb were `tsconfig.json` (`jsx: react-jsx`, dev-types path) and the dev bundler default (Turbopack — caused no behavior change in our stubs but is dramatically faster). When Phase 2/3 add real data-fetching routes, expect to deal with: (a) `fetch()` no longer caches by default — explicit `cache: 'force-cache'` if you want static caching, or use the `'use cache'` directive at the top of the function; we don't fetch HTTP yet so it's fine. (b) `searchParams: Promise<…>` is enforced (we don't use it yet). (c) Route-segment config exports (`dynamic`, `revalidate`, etc.) work the same.
- **Peer-deps intentionally NOT bumped.**
  - `react / react-dom 19.2.5` — current; satisfies Next 16's `react ≥19` peer. No reason to bump.
  - `typescript 5.9.3` — current within 5.x; Next 16 only requires `≥3.3.1`. TS 6 hasn't shipped.
  - `eslint 9.39.4` — current within 9.x; eslint-config-next 16's peer is `≥9`. ESLint 10 is in pre-release.
  - `drizzle-orm 0.45.2 / drizzle-kit 0.31.10` — current; brief explicitly out-of-scope. The 0.45 → 0.46 hop is queued in the Phase 1 report's "For next session"; defer.
  - `@supabase/supabase-js 2.105.1`, `postgres 3.4.9`, `zod 3.24.1`, `clsx 2.1.1` — current. No bumps needed; the brief flags zod 4 + the legacy-key migration as later sub-briefs.
- **Trivial cleanup waiting for Phase 2 to start:** the v3 border-color compat block in `globals.css` (lines 77–85). Will delete in the first Phase 2 commit that ports prototype CSS once we've audited every element's intended border color.
