# Timed preview access — per-person signed invite links

> **Revised 2026-06-20 after two external review passes.** Folded in from the **first pass**: the secret-unset degrade is now fully specified (it no longer contradicts the single-cookie model); domain separation (`typ`) between invite token and session cookie; the canonical token format must be written down in the report as SSOT; the console is served on `localhost` by default (dropping the `file://` / `null`-origin path) which also resolves the Web-Crypto secure-context question; the read endpoint must self-gate from its `Authorization` header (the proxy is bypassed for it, so `getIsAdmin()` won't work); token-in-URL leakage acknowledged + `Referrer-Policy` on `/login`; custom-HMAC confirmed over JWT (no new dependency) as a conscious tradeoff; open-question #1 downgraded — `crypto.subtle` is confirmed available in both runtimes. Folded in from the **second pass (Codex)**: corrected the Vercel-preview scope claim (preview deployments are **gated, not bypassed** — they build with `NODE_ENV=production`); the `OPTIONS` preflight must return CORS headers **without** requiring `Authorization` (only the `GET` is gated); the console's reviewable source is a **committed, secret-free generator/template** that produces the gitignored local output (a purely gitignored file can't be code-reviewed); and this brief was **renumbered 162 → 163** (162 collided with `2026-06-19-162-impl-entity-isr-hot-subset`).

## Goal

Let Philipp share the site with individual people over a rolling outreach (write one Reddit contact today, another four days later) by handing each one a **signed invite link** that grants a time-limited preview — each link carrying its own expiry, minted on demand. No per-user accounts; the only server-side state is **one tiny table that records link activations**, so the local console can show what's actually been used. One master secret signs everything; rotating it ends **every** outstanding link and session at once, which is the only revocation lever needed.

This **supersedes the global-window design** that was in an earlier draft of this brief — a single shared cutoff doesn't fit rolling outreach. The model is now: mint a link per person, each with an embedded absolute expiry (default ~3 days, configurable at mint).

## Design freedom — read before everything else

The invite link **routes to the login page**, reusing its existing console chrome (`src/app/login/page.tsx`) — Philipp specifically wants to keep that branded login experience rather than an invisible auto-redirect. The page gets an **invite state**: where the username/password fields normally sit, it shows a single accept action instead (Philipp's words: an "Accept invitation" button). Clicking it redeems the link and lets the person in. Three states share the one console:

- **Invite state** — arrived via a valid, unexpired link: show the accept action in place of the credential form. (The maintainer's normal username/password form is the default state when there's no invite.)
- **Invalid / expired link state** — arrived via a bad or expired link: a graceful "this invitation is invalid or has expired" affordance, distinct from the wrong-password error. It **must include a contact prompt** in Philipp's intent — "if you need another preview link, contact me" — where **"me" is a link to `https://www.reddit.com/user/piwireddit/`** (open in a new tab, `rel="noopener noreferrer"`). The URL is pinned; the exact wording and styling of the message are yours.
- The existing wrong-password error state stays as is.

Everything aesthetic is yours: the exact button copy and the "expired link" copy/voice (the page speaks an in-universe "RESTRICTED ARCHIVE / sealed" register — match or evolve it), markup, class shapes, styling, motion, and the route shape that carries the invite (a `?invite=` param on `/login` vs. a sibling route — your call). Nothing worded in this brief is final copy; treat "Accept invitation" as the maintainer's intent, not a pinned string.

Architectural requirement is only: **the accept action only appears for a server-validated, unexpired link; clicking it grants access; an invalid or expired link cannot grant access and shows the error state instead.** Look, copy, and route shape are your call.

## Context

The preview gate (Session 145, hardened 147/150) is the whole-site lock anonymous visitors hit:

- `src/lib/previewGate.ts` — shared constants: single shared credential (`PREVIEW_USER` / `PREVIEW_PASS`, committed defaults as a soft lock), `PREVIEW_COOKIE = "cl-preview"`, `previewGateEnabled()` (prod-only; `PREVIEW_GATE=off` is the launch kill-switch).
- `src/proxy.ts` — the route gate (Next 16 `proxy.ts`, ex-`middleware.ts`). On any matched route, if `previewGateEnabled()` and the `cl-preview` cookie isn't `"1"`, it 307s to `/login`. The Basic-Auth admin block below (`/ingest`, `/buch/*/audit`, `x-atlas-admin`) is **unrelated** and stays untouched.
- `src/app/login/actions.ts` — the `login` server action. On correct creds it sets `cl-preview = "1"`, `maxAge` 30 days, redirects to `/`.
- `src/app/login/page.tsx` — the login console.

The gap: the cookie value is a dumb `"1"` with no embedded expiry, and the gate only checks presence. There's no way to time-limit a visitor, and the only entry credential is one shared password.

The new shape (otherwise stateless; one tiny activations table is the sole server-side state):

1. **A signing secret** (new env var, `PREVIEW_*` family) signs both invite links and the session cookie. Set in Vercel (prod) and in Philipp's local `.env.local` (so minting and verifying agree).
2. **A single local management console** — one self-contained **HTML artifact** (committed, secret-free generator/template → gitignored local output; see Constraints + Open questions) that Philipp opens on his own machine and that is the *only* place he manages links. It does two things:
   - **Generate:** he enters a duration (default ~3 days) and an optional label and clicks a **Generate** button; the page computes `exp`, builds the signed token **in the browser** (Web Crypto HMAC over the same token format the server verifies), assembles the full absolute invite URL against the prod base, **adds it to the on-page list**, and offers a copy-to-clipboard. No command line, no separate mint step.
   - **Overview:** the list shows every link he's generated — label, **its short id (`jti`)**, minted date, expiry, the URL, and a status badge (**active vs. expired computed in-browser from `exp`**, re-evaluated on refresh). The list **persists locally** (e.g. `localStorage`) so it survives reopening the file. The `jti` is the join key between a local link and its server-side activation record.
   - **Activation overlay:** on load (and via a Refresh action) the console **fetches activation status from an admin-gated read endpoint** on the site and overlays an "activated · <first time> (×count)" badge on the links it knows about, matched by `jti`. If the fetch fails (offline, missing credential), it shows the links without the overlay plus a quiet "couldn't load activation status" hint — never a hard error.

   The signing secret **and the admin read-credential** are **entered at runtime** (and may be remembered locally), **never hardcoded into the file**. The console is operator-only: it lives on Philipp's machine, is gitignored, and is **never deployed**. It surfaces *activation* status (a link was opened+accepted, and when); what it still cannot show is whose session is *currently live* moment-to-moment.
3. **Redemption (via the login page):** the invite link routes to the login page carrying the token. The page **server-validates** the token (signature + `typ === "invite"` + `exp` in the future) before rendering: valid → it shows the "Accept invitation" state; invalid/expired/wrong-`typ` → the error state, no accept action. On clicking Accept, a server action **re-verifies** the token authoritatively and sets a **signed `cl-preview` cookie** carrying that same `exp`, then redirects to `/`. Because `exp` is absolute and baked at mint, re-clicking the link cannot extend access past it. On a successful Accept the server **records the activation** — a best-effort upsert into a tiny `preview_invite_activations` table keyed by `jti` (first/last activation time + a count). **Best-effort means:** if the write fails, the server logs the error, still sets the cookie, and lets the person in — activation tracking must never block access. The local console reads this table back through an admin-gated endpoint (point 2's overlay).
4. **The activation read endpoint** — a small admin-gated, read-only API (e.g. `GET /api/preview-invites`) returns the activation rows (`jti`, first/last time, count) as JSON for the console. It is **not** public: it requires an admin credential **checked by the route handler itself** (because the route is excluded from the proxy matcher, the proxy never runs for it — see Constraints), must be excluded from the preview-cookie redirect, and must answer the local console's cross-origin request. The console is served on `localhost` (see below), so CORS allow-origin is a concrete `http://localhost:<port>` origin plus an `OPTIONS` preflight and the auth header — **not** a `null` origin.
5. **The gate** (`src/proxy.ts`) verifies the signed cookie's signature, that `typ === "session"`, and that `exp > now` on every request; invalid/expired/absent/wrong-`typ` → `/login`. This is what evicts expired sessions and rejects tampered cookies. (Secret-unset fallback: legacy presence check — see Constraints → "Degrade, don't crash".)
6. **Kill-all lever:** rotating the signing secret in Vercel instantly invalidates every outstanding link and every active cookie (they no longer verify). That is the "end every link prematurely" control Philipp asked for.

The maintainer's own password login (`PREVIEW_USER`/`PREVIEW_PASS`) stays as a convenience entry for Philipp — but **when the signing secret is set**, it mints the **same signed cookie format** the gate verifies (with a longer expiry of its own), so there is exactly one cookie shape to verify. **When the signing secret is *unset*** the system degrades to pre-163 behaviour (see Constraints → "Degrade, don't crash"): the password login sets the legacy unsigned `cl-preview = "1"` cookie, the gate falls back to the legacy presence check, and only invite redemption is disabled. So "exactly one cookie shape" holds whenever the feature is actually configured; the unsigned legacy shape exists only as the secret-unset fallback.

## Constraints

- **New signing-secret env var** in the `PREVIEW_*` family (e.g. `PREVIEW_INVITE_SECRET`). High-entropy, **never committed** (unlike the soft-lock password — this one is load-bearing). Document it in `.env.example` as required-in-prod with a "rotate to revoke everything" note.
- **Token + cookie are HMAC-signed** over a payload that includes an absolute `exp` (epoch). Use a keyed signature with a constant-time compare (a `timingSafeEqual`-style check already exists at `src/lib/timingSafeEqual.ts`). No new dependency — use the platform's built-in crypto.
- **Verification must run in the gate's runtime.** `src/proxy.ts` is the enforcement point. Use Web Crypto `crypto.subtle` HMAC — it is confirmed available in **both** the proxy runtime and Node route handlers (see `src/lib/timingSafeEqual.ts`, which depends on exactly that), so one primitive serves the gate, the redemption handler, and the browser console alike. CC should still note in the report which runtime the proxy runs on, but no longer needs to research whether a primitive exists.
- **The gate is the enforcement point**, not just redemption: **when the secret is set**, a request whose `cl-preview` cookie fails signature **or** is past `exp` is treated as unauthenticated and redirected to `/login`. Existing pre-deploy `cl-preview = "1"` cookies therefore stop validating — current testers simply re-enter via password or a fresh link (acceptable; note it). **When the secret is unset** the gate uses the legacy presence check (`cl-preview === "1"`) instead — see the degrade constraint below. The gate must **fail closed on a configured-but-bad cookie** (signature/`exp` failure → redirect), never fail open.
- **Cookie `maxAge` must not exceed time-until-`exp`.** httpOnly, `secure` in prod, `sameSite: lax`, `path: /` (as today).
- **The management console is a single self-contained HTML artifact** (inline CSS/JS, no external CDN/library dependency — its only network call is the cross-origin read to the admin activation endpoint), produced from a **committed, secret-free generator/template** (the source of truth that gets reviewed) into a **gitignored, never-deployed** local output Philipp opens (see Open questions for the decided split). It is **served on `localhost`** (a small static-serve command, not opened as `file://`) so `crypto.subtle` has a secure context and the endpoint can use a concrete localhost CORS origin. It both generates signed links (Generate button) and lists them with live active/expired status, overlaying activation status fetched from the admin endpoint. The link list persists client-side (`localStorage` or equivalent). Add the file (and any exported list) to `.gitignore`. The console itself is never a deployed route; it only *reads* the admin-gated activation endpoint.
- **Token payload carries a short unique id (`jti`) and a type tag (`typ`) alongside `exp`.** `jti` need not be secret; it is the join key between a generated link and its activation row. Keep the human label *out* of the token (it's decodable by the recipient) — the label↔`jti` mapping lives only in the local console.
- **Domain separation between the invite token and the session cookie is mandatory.** Both payloads share the same key and would otherwise be byte-identical structures (`{exp, jti}`), so an invite-link token string would verify *as a session cookie* — a recipient could paste their invite token straight into the `cl-preview` cookie and skip the Accept step (and its activation write). Prevent this: the HMAC must bind a context, via a `typ` field in the payload (`"invite"` vs `"session"`) **that is part of the signed bytes**, not just decoration. The gate accepts only `typ: "session"` cookies; redemption accepts only `typ: "invite"` tokens. An invite token and a session cookie are then non-interchangeable even though they share a key. (Harm today is low — activation is best-effort observability — but it's a free correctness win and CC should not have to rediscover it.)
- **Activation tracking = one tiny table + one admin-gated read endpoint.** New table (Drizzle schema + a generated, committed migration), minimal and PII-free: `jti` (primary key), first-activated-at, last-activated-at, activation count. On a successful Accept the redemption action **upserts** by `jti`. The write is **best-effort** — a failure is logged and must not block the cookie being set or the redirect. The read endpoint (`GET`, admin-gated, read-only) returns these rows as JSON for the console; it must be excluded from the preview-cookie redirect, **check the admin credential in the route handler itself** (the proxy is bypassed for this path — see below), and send the CORS headers the local console needs: allow-origin is the **`http://localhost:<port>` origin** the console is served from, plus the `OPTIONS` preflight and the auth header. **Do not allow a `null` origin** — that would also admit `file://` pages and sandboxed/opaque-origin frames, an anti-pattern especially next to an `Authorization` header. **The `OPTIONS` preflight must return the CORS headers *without* requiring `Authorization`** (browsers never send credentials on a preflight — gating it on auth breaks CORS); only the actual `GET` requires the admin credential. No IP, user-agent, or other PII is stored.
- **Activation recording must not gate access.** The redemption path sets the cookie and redirects regardless of whether the DB write succeeds; the table is observability, not an authorization input.
- **One token format, two implementations that must round-trip — and the format is written down as SSOT.** The signed-token scheme is the single source of truth, but a brief that only *says* "defined once" without defining it leaves the byte-level details (base64url padding, JSON key ordering, string encoding) to chance — and that is exactly where a browser-generator-vs-server-verifier mismatch hides. So: **CC must record the canonical token spec explicitly in the implementation report** — e.g. `base64url(JSON payload, keys in fixed order) + "." + base64url(HMAC-SHA256(those bytes))`, naming the key order, the padding convention, and the encoding — as the documented SSOT both implementations are written against. The browser generator (console) and the server verifier (redemption + gate) are separate implementations of that spec and **must agree byte-for-byte** — a token the console produces must verify server-side, and vice versa. CC must add a round-trip check (console-produced token → server-verify, and back) as a regression guard.
- **Crypto primitive: custom HMAC, deliberately — not a JWT library.** Decided with Philipp this session (post-review). The payload is two-plus-fields (`typ`, `exp`, `jti`); a hand-rolled HMAC-SHA256 over a canonical encoding is honest and dependency-free, and the project's "don't add a dependency without a session-log justification" rule favours it. The conventional alternative — JWT HS256 via a vetted library — would collapse the two implementations into one well-tested one and remove the byte-for-byte-agreement risk entirely; it was **considered and declined** because it adds a dependency for a payload this small, and because the round-trip test + the written-down canonical spec above contain the risk. The conscious tradeoff: **we own the crypto correctness.** Use the platform's built-in `crypto.subtle` HMAC (confirmed available in both runtimes — see Open questions); no new dependency.
- **The signing secret never ships in the file.** The console reads the secret at runtime (entered by Philipp, optionally remembered in `localStorage`); it is never hardcoded into the committed-or-generated HTML, and the file is gitignored and never deployed. The secret's trust level is the same as in `.env.local`.
- **Enforcement scope unchanged — and note what "the gate already applies" actually means.** Window/link logic only applies where the gate already applies: `previewGateEnabled()` is `NODE_ENV === "production" && PREVIEW_GATE !== "off"` (`src/lib/previewGate.ts`). **Local dev bypasses** (`NODE_ENV !== "production"`). **Vercel *preview* deployments are gated, NOT bypassed** — they build with `NODE_ENV === "production"`, so the gate is live on them, which Session 145 chose deliberately so PR previews can exercise the gate. Do **not** "fix" this into a preview-bypass; the new invite/cookie logic must inherit the exact same scope as the existing gate. `PREVIEW_GATE=off` still disables the whole gate for launch.
- **Degrade, don't crash — and the degrade path is *specified*, not left to CC's guess.** "Exactly one cookie shape" is true only while the secret is set; the unset case has its own posture, and CC must be told it explicitly so they neither fail the gate open (defeats it) nor lock the site out (every visitor loops to `/login`). **When the signing secret is unset in prod, the whole feature degrades to pre-163 behaviour:**
  - The **gate** falls back to the legacy presence check — `cl-preview === "1"` grants access; it does **not** try to verify a signature it has no key for, and it does **not** fail open to everyone.
  - The **password login** sets the legacy *unsigned* `cl-preview = "1"` cookie (the pre-163 shape), so Philipp can still get in.
  - **Invite redemption is disabled** and the server logs an error; an invite link in this state simply shows the invalid/expired state.
  - The **activation endpoint** still answers (it has no dependency on the signing secret) but there will be nothing new to record.

  This matches the existing `ATLAS_PASS`-missing posture — log + degrade, never take the public site down over a misconfigured secret. The acceptance bullet for the unset case asserts exactly this fallback (gate works via presence check, invites off), not a vague "still gates somehow".
- **The only persistence is the tiny activations table; no real accounts, no per-user denylist, no new dependency.** Revocation stays rotate-the-secret-kills-all, by the maintainer's choice.
- **Token-in-URL leakage — acknowledged and bounded.** The invite token rides in the `?invite=` query param, so it lands in Vercel access logs and the recipient's browser history. For a friendly fan-site preview with short-lived, single-purpose tokens this is an **accepted** tradeoff (it's the price of a paste-a-link flow with no accounts). Bound it: set a `Referrer-Policy` (e.g. `no-referrer` or `strict-origin`) on the `/login` response so the token can't leak via the `Referer` header if an outbound link is ever rendered in the valid/invite/expired states (the "contact me" Reddit link in particular). One header, not a redesign.
- **TypeScript strict, no `any`.** All signing/verification is server-only; the secret never reaches a client component.

## Out of scope

- The Basic-Auth **admin** block in `src/proxy.ts` (`/ingest`, `/buch/*/audit`, `x-atlas-admin`, `getIsAdmin()` plumbing) — do not touch its existing logic. **Exception:** you may extend the proxy matcher's exclusions so the new read endpoint bypasses the preview-cookie redirect. **But note the consequence:** once `/api/preview-invites` is excluded from the matcher, the proxy never runs for it, so the route **cannot** lean on `x-atlas-admin` / `getIsAdmin()` (those are set *by* the proxy). The **route handler must check the admin credential from the `Authorization` header itself** — reuse the existing admin credential (don't invent a parallel scheme), but verify it in-handler. Wiring it to `getIsAdmin()` would yield an always-unauthenticated endpoint; that is the trap to avoid.
- The `PREVIEW_GATE=off` launch kill-switch semantics — leave intact.
- **Per-person revocation / denylist** — out of scope; the kill lever stays rotate-the-secret-kills-all. (Activation *tracking* is now in scope via the tiny table, but using it to revoke a single link — a denylist check in the gate — is not.)
- Real user accounts, the `submissions` table, or any account/identity model. The activations table stores only `jti` + timestamps + count, never identity.
- Renaming the `cl-preview` cookie or changing `PREVIEW_USER` / `PREVIEW_PASS` values.
- The open-questions queue (16b/c timeline follow-ups, OQ 18 `db:apply` deepening) stays deferred to the next board session — this is an out-of-band maintainer request, not board work.

## Open questions

- **Runtime of `src/proxy.ts`** (edge vs. node) — *largely resolved.* `crypto.subtle` is confirmed available in **both** the proxy runtime and Node route handlers (documented in `src/lib/timingSafeEqual.ts`, which relies on exactly that), so the cookie verification can use Web Crypto HMAC in the gate directly. CC should still confirm and note in the report which runtime the proxy actually runs on. If verifying in the proxy ever proves awkward, a thin verified re-check in a server boundary is acceptable as long as no gated route renders for an unverified/expired cookie.
- **The console is served on `localhost`, not opened as `file://` — this is now the decided default, not a fallback.** Two reasons converge on it: (a) `crypto.subtle` requires a secure context and is unreliable on `file://`, while `localhost` *is* a secure context; (b) it lets the activation endpoint use a concrete `http://localhost:<port>` CORS origin instead of the `null`-origin anti-pattern. So provide a tiny static-serve path (e.g. an `npm run preview:console` that serves the gitignored console file on a fixed `localhost` port) and document it. The `file://` path and any `null`-origin handling are **dropped**. Note the exact command and port in the report so the endpoint's allow-origin and Philipp's open-the-console step match.
- **Where the console file lives — decided: a committed, secret-free generator/template produces the gitignored local HTML.** A *purely* gitignored HTML file can't be code-reviewed or reproduced by CI, so the source of truth must be committed: a committed template/generator (no secrets in it — the secret and admin credential are entered at runtime, never baked in) that produces the gitignored, never-deployed console artifact Philipp actually opens. CC owns the generation mechanics (a template file + a tiny build step, an `npm run` script, whatever's cleanest) — but the reviewable, committed source + gitignored output split is the requirement, not optional. The gate/login code is Product-strand; keep the whole feature on one task branch / one PR (the file set spans `src/**` + the committed console source). - Default link duration: I've said ~3 days, overridable in the console. Adjust the default if you think another value is the better baseline and note it.
- **Admin credential for the read endpoint.** Decide whether the console authenticates to `/api/preview-invites` by reusing the existing admin Basic-Auth (`ATLAS_USER`/`ATLAS_PASS`) via an `Authorization` header, or a dedicated read token.
