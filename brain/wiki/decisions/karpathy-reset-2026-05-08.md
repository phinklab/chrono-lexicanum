# Karpathy-Reset — Brain + Atlas

## Context

By 2026-05-08 Phase 3 had produced more material than Cowork's session-start could digest from raw logs: ~50 sessions, ~21 ingestion modules, two external code reviews, a 9-item Carry-over (against an aspirational limit of 3–5). Top-level docs had drifted post-Plan-Reshuffle 2026-05-02. There was no editable home for engineering memory that sat between "raw session log" and "the code itself."

Karpathy's [LLM Wiki gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) (with [Starmorph's conventions](https://blog.starmorph.com/blog/karpathy-llm-wiki-knowledge-base-guide)) provides exactly that home: synthesized wiki pages against immutable raw sources, three operations Ingest / Query / Lint. The project-specific twist: book-domain content (~800 books × N fields) cannot share the always-loaded Brain context with engineering knowledge — token-vorteil collapses. Two stores, hard split.

## Decision

**Brain (`brain/` in repo) + Atlas (external Obsidian vault `chrono-atlas/`, default `~/chrono-atlas/`).**

- Brain follows Karpathy + Starmorph: `brain/CLAUDE.md` schema · `brain/wiki/` synthesized · `brain/raw/` immutable · `brain/outputs/` generated. Always auto-loaded via top-level CLAUDE.md.
- Atlas is **not** a Karpathy wiki. Mechanical mirror of Postgres, regenerated by `npm run atlas:regen`. Never auto-loaded; only opened on explicit "schau in den Atlas".
- Top-level docs reshape: `README.md` slim · `ARCHITECTURE.md` / `ROADMAP.md` / `ONBOARDING.md` → 4-line redirects to brain wiki · top-level `CLAUDE.md` keeps stack/conventions + `## Brain & Atlas` section.

## Why

- **Atomic commits.** Code change + wiki update land together; reviewers see them. External-vault Brain breaks this.
- **Auto-load.** Top-level CLAUDE.md is loaded by the harness on every session. Brain in repo means no separate sync discipline.
- **Domain split mirrors physical split.** External Atlas serves visualization (Obsidian graph view); in-repo Brain serves engineering memory (VS Code editing). Different jobs, different tools.
- **Manual atlas-regen.** 3d-Apply doesn't exist yet, Atlas is allowed to be stale (Postgres is canonical), no coupling, no race conditions.

## Revisit triggers

- Wiki grows past ~50 pages or ~2 MB and Query feels slow → consider [QMD](https://github.com/tobi/qmd) (Karpathy-recommended BM25/Vector/LLM-rerank) as a search layer.
- Manual atlas-regen gets forgotten before screenshots / Reddit-launch → wire an auto-trigger after 3d-Apply.
- A third domain emerges (e.g. licensing knowledge from Black Library negotiations) that fits neither store → consider whether it deserves a third store, but be skeptical (every new store dilutes the discipline).
- Two-vault setup proves too high-friction for solo-Philipp use → consider folding Atlas back into Brain with `wiki/atlas/` subfolder + `tags: [book-data]` selective loading. Today the split is non-negotiable; revisit after 6 months.
